# AGENTS.md

## 專案定位

這是一個「租賃車／接送車資試算與預約網站」。

主要用途是讓使用者輸入出發地、目的地、乘車時間、車型、人數、行李與加購項目，系統協助計算接送車資，並產生預約資料給客服或後端保存。

目前重點不是重做 UI，而是讓「地點判斷、開車距離、車資計算、訂單送出」變得可靠。

## 服務類型

網站目前支援這幾種接送情境：

- 預約送機：從指定地點前往機場
- 預約接機：從機場前往指定地點
- 市區單程接送：一般點到點接送
- 當日臨時預約：短時間內的臨時派車需求，不保證一定有車

請保留這些情境，不要任意刪除。

## 開發原則

- 優先修正商業邏輯，不要先大改畫面。
- 保留目前深色、金色、高級感的視覺風格。
- 不要把報價邏輯寫死在前端。
- 不要讓前端決定最終價格。
- 不要讓使用者只輸入文字地址就直接報價。
- 地點名稱有歧義時，例如「麗山國中」，必須顯示候選地點讓使用者確認。
- 正式報價應使用 Google Maps 的開車距離，不要用直線距離推估正式價格。
- 若無法取得可靠路線，應顯示「需人工確認」，不要產生假報價。

## 地點處理規則

使用者輸入地點時，系統應該先做地點搜尋，讓使用者選擇正確地點。

正確流程：

1. 使用者輸入地點名稱。
2. 系統查詢地點候選結果。
3. 前端顯示地點名稱與地址。
4. 使用者選定其中一個地點。
5. 系統保存該地點的 `place_id`、名稱、地址與座標。
6. 報價時使用已確認的地點資料。

不要直接相信第一筆搜尋結果。

## 距離計算規則

正式報價應使用開車距離。

建議流程：

1. 起點與終點都必須是已確認地點。
2. 後端使用 Google Routes API 計算開車距離。
3. 取得 `distanceMeters`。
4. 後端用距離與價目表計算價格。
5. 前端只顯示後端回傳的報價。

禁止把 Haversine 直線距離當作正式報價依據。

## 車資計算邏輯

目前基本邏輯先保留：

### 機場接送

若符合機場接送價目表，優先使用固定價。

例如：

- 台北市到桃園機場
- 汐止區到桃園機場
- 文山區到桃園機場
- 台北市到松山機場

價目表先放在後端設定檔，之後再考慮搬到資料庫。

### 一般距離計價

若沒有符合固定價目表，使用距離計價：

```text
base_price = round(distance_km * 20)
minimum_price = 799
final_base_price = max(base_price, minimum_price)
```

### 車型加價

目前先保留：

```text
一般舒適四人座：+0
尊榮進口車：+500
七人座商旅：+300
七人座尊爵：+2000
```

### 加購項目

目前先保留：

```text
專人舉牌：+200
兒童安全座椅／增高墊：+300
```

### 最終價格

```text
total_price = base_price + vehicle_surcharge + addon_price
```

最終價格必須由後端計算。

## 安全規則

- Google API key 不可以放在前端。
- Google Sheet webhook 不可以放在前端。
- `.env` 不可以 commit。
- 前端送來的價格不可信。
- 後端必須重新計算或驗證報價。
- 訂單成立前，必須使用後端報價結果。

## 建議後端 API

可以先用 Python FastAPI 實作。

### `POST /api/places/search`

用途：搜尋地點候選結果。

輸入：

```json
{
  "query": "麗山國中"
}
```

輸出：

```json
{
  "candidates": [
    {
      "place_id": "google_place_id",
      "name": "臺北市立麗山國民中學",
      "address": "台北市內湖區...",
      "lat": 25.0,
      "lng": 121.0
    }
  ]
}
```

### `POST /api/quote`

用途：計算接送報價。

輸入：

```json
{
  "service_type": "airport-send",
  "origin_place_id": "origin_place_id",
  "destination_place_id": "destination_place_id",
  "vehicle_type": "business_7",
  "passengers": 2,
  "luggage": 3,
  "addons": {
    "sign": false,
    "child_seat": true
  }
}
```

輸出：

```json
{
  "quote_id": "Q202606240001",
  "distance_km": 43.1,
  "base_price": 1400,
  "vehicle_surcharge": 300,
  "addon_price": 300,
  "total_price": 2000,
  "pricing_method": "airport_zone_table"
}
```

### `POST /api/orders`

用途：確認預約並建立訂單。

輸入：

```json
{
  "quote_id": "Q202606240001",
  "customer_name": "陳先生",
  "phone": "0912345678",
  "email": "test@example.com",
  "notes": "需要三聯式發票"
}
```

輸出：

```json
{
  "order_id": "O202606240001",
  "status": "created"
}
```

## 不要做的事

- 不要一開始就重寫整個網站。
- 不要把專案改成過度複雜的架構。
- 不要在前端寫死正式報價。
- 不要自動選第一個地點搜尋結果。
- 不要在路線失敗時用直線距離假裝成功。
- 不要暴露任何後端金鑰或 webhook。

## 優先開發順序

1. 保留現有 UI。
2. 把地點搜尋改成可選候選地點。
3. 把距離計算搬到後端。
4. 把報價邏輯搬到後端。
5. 把訂單送出搬到後端。
6. 最後才考慮重構前端框架。
